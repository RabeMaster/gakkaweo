# 커밋 컨벤션

이 프로젝트는 [Conventional Commits](https://www.conventionalcommits.org/ko) 규칙을 따릅니다.

## 커밋 메시지 형식

```
타입: 제목

본문 (선택)
```

### 제목

- **타입**은 아래 표에서 선택합니다.
- 한글로 간결하게 작성하며, 마침표를 붙이지 않습니다.
- 명령문으로 작성합니다. (`추가`, `수정`, `변경` 등)
- 첫 글자는 소문자로 시작합니다.

### 본문

- 제목만으로 설명이 부족한 경우 작성합니다.
- 제목과 본문 사이에 빈 줄을 넣습니다.
- **무엇을** 변경했는지보다 **왜** 변경했는지를 중심으로 작성합니다.
- 제목과 마찬가지로 명령문으로 작성합니다.
- 항목이 여러 개일 경우 `-`(하이폰)으로 나열합니다.

### 예시

```
feat: 소셜 로그인 기능 추가

- 기존 이메일 로그인 외에 Google, Kakao OAuth 로그인 지원
- passport 라이브러리를 사용하여 구현
```

## 타입 목록

| 타입 | 설명 | 예시 |
|------|------|------|
| feat | 새로운 기능 추가 | `feat: 사용자 로그인 기능 추가` |
| fix | 버그 수정 | `fix: 토큰 만료 시 무한 루프 수정` |
| docs | 문서 추가/수정 | `docs: API 문서 업데이트` |
| style | 코드 포맷팅, 세미콜론 누락 등 (동작 변경 없음) | `style: 들여쓰기 통일` |
| refactor | 기능/버그 변경 없이 코드 구조 개선 | `refactor: 인증 로직 모듈 분리` |
| test | 테스트 추가 또는 수정 | `test: 로그인 API 단위 테스트 추가` |
| chore | 빌드 설정, 패키지 관리 등 기타 변경 | `chore: Docker 설정 추가` |
| ci | CI/CD 파이프라인 설정 변경 | `ci: GitHub Actions 워크플로우 추가` |
| perf | 성능 개선 | `perf: 이미지 로딩 최적화` |

## 좋은 커밋 메시지 vs 나쁜 커밋 메시지

```
# Good
feat: 소셜 로그인 기능 추가
fix: 비밀번호 재설정 시 이메일 미발송 수정
refactor: 유저 서비스 레이어 분리

# Bad
fix: 수정
feat: 업데이트
chore: 작업
```

## 참고

- 하나의 커밋에는 하나의 변경 목적만 담습니다.
- 커밋 타입이 애매할 경우 가장 핵심적인 변경에 맞춰 선택합니다.
