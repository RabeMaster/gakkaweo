# 브랜치 전략

이 프로젝트는 `main` / `dev` 기반의 브랜치 전략을 사용합니다.

## 브랜치 구조

```
main          ← 배포용 (dev에서 머지, 배포 시점에만)
 └── dev      ← 개발 통합 브랜치
      ├── feat/#15-로그인-기능
      ├── fix/#22-토큰-만료-오류
      ├── docs/#30-API-문서-작성
      └── chore/#8-도커-설정
```

| 브랜치               | 용도                                        | 머지 대상      |
| -------------------- | ------------------------------------------- | -------------- |
| main                 | 배포용. 항상 안정된 상태를 유지             | -              |
| dev                  | 개발 통합. 모든 작업 브랜치가 여기로 머지됨 | main (배포 시) |
| feat/#이슈번호-설명  | 새로운 기능 개발                            | dev            |
| fix/#이슈번호-설명   | 버그 수정                                   | dev            |
| docs/#이슈번호-설명  | 문서 작업                                   | dev            |
| chore/#이슈번호-설명 | 빌드, 설정 등 기타 작업                     | dev            |

## 브랜치 네이밍 규칙

```
타입/#이슈번호-간단한-설명
```

- 이슈 번호는 GitHub 이슈 번호를 사용합니다.
- 설명은 소문자 한글 또는 영문, 단어 구분은 `-`(하이픈)으로 합니다.

### 예시

```
feat/#15-로그인-기능
fix/#22-토큰-만료-오류
docs/#30-API-문서-작성
chore/#8-도커-설정
```

## 워크플로우

1. **이슈 생성** — GitHub에서 이슈 템플릿을 사용하여 이슈를 생성합니다.
2. **브랜치 생성** — dev에서 이슈에 대응하는 브랜치를 생성합니다.
   ```bash
   git checkout dev
   git pull origin dev
   git checkout -b feat/#15-로그인-기능
   ```
3. **작업 및 커밋** — [커밋 컨벤션](commit-convention.md)에 맞춰 커밋합니다.
4. **PR 생성** — GitHub에서 dev 브랜치로 PR을 생성합니다 (PR 템플릿 사용).
5. **셀프 리뷰 및 머지** — 셀프 리뷰 후 dev에 머지합니다.
6. **배포** — 배포 시점에 dev를 main에 머지합니다.
